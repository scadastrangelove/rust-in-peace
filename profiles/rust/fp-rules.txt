RUST SECURITY — triage exclusion rules (appended to the /triage verifier).
Use with:  /triage <findings>.json --fp-rules profiles/rust/fp-rules.txt

These are FALSE_POSITIVE precedents for Rust targets. A finding matching one is
FALSE_POSITIVE even if the cited code is technically as described. Cite the rule
number in the verdict. Each rule demands the verifier do a specific TRACE, not a
pattern match.

RULE R1 — Invariant-protected unsafe read.
An `unsafe` unchecked read (`read_unaligned`, `get_unchecked`, `.add()`,
`from_raw_parts`, raw index) is a FALSE_POSITIVE if the index/offset/length it
uses is provably bounded by a checked invariant established BEFORE the read:
a preceding exclusion/mask that removes the dangerous positions, a parse-time
validation of the field, an enforced zone/record/buffer size, or a caller
contract enforced at the trust boundary. To apply this rule the verifier MUST
trace the invariant to its source and show it dominates the read on every path —
not merely assert "looks guarded." (If the invariant is only a `debug_assert!`,
R1 does NOT apply — see R4.)

RULE R2 — Operator-only / trusted-by-construction trust boundary.
A finding whose trigger requires an input that, IN THIS DEPLOYMENT, is
operator-controlled or trusted-by-construction — a compiler/toolchain-emitted
artifact, an operator-signed or CI-produced blob, an internal-only database, a
value already validated at an outer trust boundary — and is NOT reachable from
attacker-controlled data, is downgraded to latent-hardening and excluded from
the live-vulnerability set. The verifier MUST state which input carries the
finding and classify it attacker- vs operator-controlled (the
"reachable-from-untrusted-input?" question). If attacker-reachability is
plausible (e.g. multi-tenant, user-uploaded, network-sourced), R2 does NOT apply
— keep it.

RULE R3 — Memory safety in safe Rust.
A memory-corruption category (UAF, double-free, OOB, data race) reported in code
with NO `unsafe` block and NO FFI on the path is a FALSE_POSITIVE — the borrow
checker and bounds checks preclude it. (Panics / DoS in safe Rust are NOT
excluded by R3; those are real availability issues — judge them on R2/R5.)

RULE R4 — Release-stripped assertion with no consequence.
A panic or "OOB" that can only fire in a debug build (a `debug_assert!`, or an
arithmetic-overflow panic that the shipped `[profile.release]` disables) AND
whose value has no downstream memory/index/allocation/security-decision
consequence, is a FALSE_POSITIVE for the shipped artifact. BUT: if the same
release path then does an UNCHECKED index/allocation with the un-asserted value
(the assert was the only guard), it is a TRUE_POSITIVE — the stripped assert is
the bug. Check what the release build actually does after the assert.

RULE R5 — Operator-controlled panic surface.
A panic (`unwrap`/`expect`/index/`try_into().unwrap()`/overflow) reachable only
from CLI arguments, config files, environment variables, build scripts, or
`#[cfg(test)]`/bench/example code is a FALSE_POSITIVE (operator input, same
class as the standard env-var/CLI exclusion). Only panics reachable from
attacker-controlled data through a public/exported API survive.

RULE R6 — Proven-Some/Ok unwrap.
`unwrap()`/`expect()` on a value the code just constructed or proved present
on the same path (insert-then-get, `is_some()`-guarded, a literal, a checked
length) is a FALSE_POSITIVE. Only unwraps on fallible operations over untrusted
data (parse, decode, checked conversion of an untrusted number) are real.

RULE R7 — Wrong-value-only integer overflow.
An integer overflow (`wrapping_*`, unchecked `+`/`-`/`*`, `as` truncation) that
produces only an incorrect value with NO effect on a subsequent index, pointer
offset, allocation size, or security decision is a FALSE_POSITIVE. If the
wrapped value THEN indexes memory or sizes an allocation, it is a TRUE_POSITIVE
(that is the actual OOB / DoS).

GENERAL. When a finding survives all rules, keep the verifier's standard
TRUE_POSITIVE bar (reachable from untrusted input, concrete trigger). For every
verdict, record REACHABLE_IF_TRUSTED_INPUT (yes/no) so ranking can separate live
vulns from latent hardening — that field, not raw severity, usually drives the
real priority order for Rust targets.

── Calibration from real RustSec incidents (canonical review §15;
   full text: profiles/rust/references/rust-security-review.md) ──

RULE R8 — Soundness ≠ security, both count.
"safe API, unsound internals" (a safe fn whose contract safe code can break —
e.g. flatbuffers reinterpreting bytes as `bool`, tracing's `mem::forget` UAF) is
a TRUE_POSITIVE even if no current caller triggers it — it is exactly the class
that becomes exploitable after a refactor or compiler update. Conversely, a pure
security-logic bug in fully-safe Rust (rustls-webpki name-constraint accept,
hickory DNS cache poisoning, a Deno permission bypass) is ALSO a true positive:
memory safety holding does not make it a false positive. R3 (safe-Rust) excludes
only memory-corruption categories, never logic / authz / protocol bugs.

RULE R9 — "we don't call that path" is not a false positive.
An unsafe-soundness or panic finding does NOT become FALSE_POSITIVE merely
because the current code doesn't reach it, IF the API is public/exported and a
future caller could. Downgrade its *reachability/severity*, but keep it (unsafe
bugs resurface under refactors). This is distinct from R2 (operator-only *input*)
and R1 (an invariant that actually *dominates* the sink today).

RULE R10 — Panic-safety is memory safety in unsafe code.
A panic that unwinds through a temporarily-broken invariant in an `unsafe` region
(via a callback, `Drop`, or allocation) producing UB / UAF / double-free
(rkyv-style) is a real memory-safety TRUE_POSITIVE, not "just a panic" — do not
apply the availability-only lens (R5) to it.

RULE R11 — Real-but-unlabeled is NOT a false positive (converse of R2/R9).
On a labeled corpus (a benchmark, a "fixed" variant, a known-CVE sample), a flag
that does NOT match the seeded label is a FALSE_POSITIVE *only if the code is
actually safe*. If the traced finding is a genuine reachable bug that the label
simply doesn't cover — a latent bug the upstream fix never touched, a second
distinct bug at/near the seeded site, a soundness hole orthogonal to the CVE —
it is a TRUE_POSITIVE with DISPOSITION=`real_latent`, NOT an FP. The bar for
`real_latent` is the SAME as any TP: an untrusted-input trace + a concrete
trigger (or, for soundness classes, a DEFER-TO-DYNAMIC-confirmable adversarial
impl). "It isn't the seeded bug" is a labeling fact, never a safety fact —
measuring precision by counting `real_latent` as FP under-reports true precision
(the four rust-mizan "false alarms" on fixed variants were all real latent bugs;
true precision was ~100%, measured 84% only because R11 wasn't yet applied).
Every `real_latent` on a *fixed* variant is a WIN → route to the latent-review
queue for possible upstreaming; do not auto-file, do not count against precision.

── Triage output schema (labeled-corpus mode) ──

Every surviving flag on a labeled sample MUST carry:

  DISPOSITION ∈ { same_as_seeded | real_latent | false_positive }
    same_as_seeded  — matches the seeded label (site + class within the fix region)
    real_latent     — a real, reachable bug the label does NOT cover (R11); the WIN case
    false_positive  — the code is actually safe (some R1..R7 exclusion fired)

  LABEL_DELTA  (required iff DISPOSITION=real_latent) — one line tracing WHY it is
    distinct from the seed: different sink / different file / fix-untouched region /
    orthogonal class. Without a traced delta it is same_as_seeded, not real_latent.

  REACHABLE_IF_TRUSTED_INPUT ∈ { yes | no }  (kept from GENERAL, drives ranking)

Report FIVE numbers, never a single "specificity":
  1. detection-recall       — flags that hit ANY real bug / all real bugs present
  2. seeded-CVE-recall      — same_as_seeded flags / seeded CVEs
  3. precision              — TP / (TP + FP), where FP = ONLY false_positive
                              (real_latent is NEVER in the denominator)
  4. seeded-CVE precision   — same_as_seeded / (same_as_seeded + false_positive)
  5. latent-bug yield       — count of distinct real_latent (a POSITIVE result,
                              never subtracted from any score)
"0 false positives but 3 real_latent" is a strong outcome, not a mixed one.
